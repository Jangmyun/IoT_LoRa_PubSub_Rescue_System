# PRD — LoRa 경량 pub/sub 기반 해양 익수자 탐지 시스템

**프로젝트명:** `lora_pubsub_rescue`
**과목:** IoT 실습 (컴퓨터 네트워크 연계)
**팀 규모:** 2~3인
**개발 기간:** 3주
**문서 버전:** v1.0

> 상세 탐지 알고리즘, 시각화, 얕은 호수 인형 시연 범위 및 검증 절차는
> [얕은 호수 인형 시연용 위험 수면 교란 감지 설계](docs/DETECTION_ALGORITHM_DESIGN.md)를 참고한다.

---

## 1. 개요 (Overview)

소나(초음파) 센서와 가속도 센서를 탑재한 자율 부표 여러 개를 수면에 배치하고, 익수자 발생 시 부표가 수면 교란을 감지하여 LoRa 멀티홉 릴레이를 통해 안전 구역의 수신국으로 경보를 전송하는 시스템이다.

이 프로젝트의 핵심 기술 과제는 컴퓨터 네트워크 수업에서 학습한 **MQTT의 pub/sub 패러다임을 LoRa 물리 계층 위에 직접 설계·구현**하는 것이다. 표준 MQTT는 TCP와 중앙 브로커를 전제로 하지만, LoRa는 TCP가 없고 64바이트 FIFO 제약을 가지므로 이 환경에 맞는 경량 프로토콜을 새로 설계한다.

---

## 2. 문제 정의 (Problem Statement)

해수욕장·호수·항만 등에서 익수 사고는 빠른 발견이 생존율을 좌우한다. 그러나 광범위한 수면을 상시 감시하는 것은 인력만으로 한계가 있다. 기존 CCTV 기반 감시는 시야 사각지대와 야간 취약성이 있고, 통신 인프라(WiFi/셀룰러)가 닿지 않는 원거리 수면은 모니터링이 어렵다.

본 시스템은 저전력·장거리 통신이 가능한 LoRa를 활용해 인프라가 없는 수면에서도 익수자를 탐지하고, 안전 구역으로 경보를 전파하는 것을 목표로 한다.

---

## 3. 목표 (Goals) 및 비목표 (Non-Goals)

### 3.1 목표

- 초음파 센서로 수면 교란을 감지하고, 가속도 센서로 파도에 의한 오탐을 보정한다.
- LoRa 위에서 동작하는 경량 pub/sub 프로토콜을 직접 설계·구현한다.
- 부표 간 멀티홉 릴레이를 구현하여 수신국에서 먼 부표의 경보도 전달한다.
- Raspberry Pi 수신국에서 경보를 수신하고 실시간 클라이언트 대시보드로 시각화한다.
- 이벤트 기반 동작과 송신 규칙을 통해 배터리 수명을 확보한다.

### 3.2 비목표 (이번 범위에서 제외)

- 익수자의 정밀 위치 추정(TDOA 삼각측량)은 선택/확장 기능으로만 명시한다.
- 실제 해양 장기 배포(방수 내구성, 해수 부식, 태양광 충전)는 프로토타입 범위 밖이다.
- 영상 기반 인식(카메라/컴퓨터비전)은 포함하지 않는다.

## 4. 시스템 아키텍처 (System Architecture)

```
[부표 A]---LoRa---[부표 B]---LoRa---[부표 C]
   |                                    |
   +------- 멀티홉 릴레이 -------+
                  |
                  v
         [Raspberry Pi 수신국]
          - LoRa 수신 게이트웨이
          - 경량 pub/sub 브로커
          - WebSocket 서버
                  |
                  v
          [클라이언트 대시보드]
           - 실시간 경보 표시
           - 부표 상태 모니터링
```

| 구성 요소           | 역할                               | 비고                    |
| ------------------- | ---------------------------------- | ----------------------- |
| 자율 부표 (×3)      | 수면 교란 감지 + LoRa 송신/릴레이  | TTGO LoRa32 기반        |
| LoRa 메시 채널      | 부표 간 멀티홉 경보 전파           | 경량 pub/sub 프로토콜   |
| Raspberry Pi 수신국 | 경보 수신 + 브로커 + 대시보드 서버 | 안전 구역 설치          |
| 클라이언트 대시보드 | 실시간 시각화                      | 웹 브라우저 (WebSocket) |

## 5. 하드웨어 사양 (Hardware Specification)

| 부품        | 모델                         | 역할                           |
| ----------- | ---------------------------- | ------------------------------ |
| MCU + LoRa  | TTGO LoRa32 (ESP32 + SX1276) | 제어 및 통신 통합 보드         |
| 초음파 센서 | JSN-SR04T (방수형)           | 수면까지 거리 측정 → 교란 감지 |
| 가속도 센서 | GY-521 (MPU-6050)            | 부표 흔들림 측정 → 파도 보정   |
| 전원        | LiPo / 대용량 배터리         | 이벤트 기반 동작으로 수명 확보 |
| 인클로저    | 방수 공 형태 케이스          | 센서 헤드만 노출, 실리콘 밀봉  |

### 5.1 배선 핵심

- JSN-SR04T: VCC 5V, Echo 핀은 전압 분배기로 3.3V 강하 후 ESP32 GPIO 연결
- GY-521: I2C (SDA→GPIO21, SCL→GPIO22), LoRa의 SPI와 버스 분리되어 충돌 없음
- LoRa SX1276이 점유하는 SPI 핀(GPIO 5/18/19/27 등)과 센서 핀 충돌 회피

## 6. 핵심 기능 요구사항 (Functional Requirements)

### 6.1 익수자 탐지 (센서 퓨전)

- **FR-1.1** 초음파 센서로 수면까지 거리를 주기적으로 샘플링한다.
- **FR-1.2** 거리값의 분산(또는 표준편차)이 임계값을 초과하면 교란으로 판정한다.
- **FR-1.3** 가속도 센서로 부표 자체의 흔들림을 측정한다.
- **FR-1.4** 거리값 변동이 크지만 부표 자체 흔들림은 작은 경우(= 외부 교란)에만 익수자로 판정한다. 부표가 함께 크게 흔들리면 파도로 판정하여 오탐을 억제한다.
- **FR-1.5** 판정 로직은 임계값 기반 룰(FFT 주파수 분석 + 분산 임계값)로 구현한다.

### 6.2 LoRa 경량 pub/sub 프로토콜

- **FR-2.1** LoRa 위에서 동작하는 토픽 기반 pub/sub 패킷 포맷을 직접 정의한다.
- **FR-2.2** 패킷 구조: `[version | type | topic_len | topic | payload_len | payload | crc]`
- **FR-2.3** 메시지 타입: PUBLISH, SUBSCRIBE, ACK, RELAY 등을 정의한다.
- **FR-2.4** 64바이트 FIFO 제약 안에 토픽과 페이로드가 들어가도록 설계한다.
- **FR-2.5** 브로커 없이 브로드캐스트 기반으로 pub/sub을 구현한다.

### 6.3 멀티홉 릴레이

- **FR-3.1** 수신국에서 직접 닿지 않는 먼 부표의 경보를 중간 부표가 릴레이한다.
- **FR-3.2** 패킷에 TTL(또는 hop count) 필드를 두어 무한 릴레이를 방지한다.
- **FR-3.3** 중복 패킷 수신 시 메시지 ID 기반으로 중복 전파를 억제한다.

### 6.4 수신국 및 대시보드

- **FR-4.1** Raspberry Pi가 LoRa 패킷을 수신하고 해당 토픽을 구독 처리한다.
- **FR-4.2** 수신한 경보를 WebSocket으로 클라이언트에 실시간 전달한다.
- **FR-4.3** 대시보드는 각 부표의 상태(정상/경보)와 마지막 수신 시각을 표시한다.
- **FR-4.4** 경보 발생 시 화면에 시각적 알림(팝업/색상 변화)을 표시한다.

### 6.5 전력 관리

- **FR-5.1** 센싱과 LoRa 송신은 일정한 규칙(주기적 샘플링, 이벤트 시에만 송신)으로 동작한다.
- **FR-5.2** 평상시에는 짧은 주기로 센서만 측정하고, 교란 감지 시에만 LoRa를 송신하여 평균 소비 전류를 낮춘다.
- **FR-5.3** 송신 빈도와 샘플링 주기는 설정 가능한 파라미터로 둔다.

## 7. 비기능 요구사항 (Non-Functional Requirements)

- **NFR-1 (신뢰성)** 경보 패킷은 멀티홉을 거쳐도 수신국에 도달해야 하며, 중복은 억제되어야 한다.
- **NFR-2 (전력)** 시연 환경에서 최소 수일 이상 연속 운용이 가능해야 한다.
- **NFR-3 (확장성)** 부표 수를 늘려도 토픽 구조와 릴레이 로직이 동작해야 한다.
- **NFR-4 (제약 준수)** 모든 패킷은 LoRa 64바이트 FIFO 제약을 만족해야 한다.

## 8. 선택 / 확장 기능 (Optional / Future)

- **TDOA 위치 추정:** 여러 부표의 탐지 타임스탬프 차이로 익수자 위치를 삼각측량한다. 각 부표가 pub/sub으로 정확한 타임스탬프를 전달하는 것과 직결되며, 대시보드 지도에 추정 위치를 표시할 수 있다.
- **저전력 MCU 전환:** 장기 배포 시 딥슬립 전류가 낮은 MCU로 교체.
- **태양광 충전:** 부표 상단 솔라 패널로 상시 운용.

## 9. 기술적 어필 포인트 (Technical Highlights)

1. **LoRa 위 경량 pub/sub 직접 설계** — 기성 라이브러리가 아닌, TCP·브로커 없는 환경에 맞춰 프로토콜 레이어를 직접 제작. MQTT가 왜 그렇게 설계되었는지를 제약 환경에서 재해석.
2. **멀티홉 릴레이** — 단순 점대점을 넘어, 수업에서 배운 라우팅·TTL·중복 억제 개념을 실제로 구현.
3. **초음파 + 가속도 센서 퓨전** — 단일 센서의 한계(파도와 익수자 구분 불가)를 두 신호 결합으로 극복.

### 설계 변경 사항 (PRD FR-2.2 대비)

PRD의 원안 `[version | type | topic_len | topic | payload_len | payload | crc]` 대신 아래 구조로 확정하였다.

```
[preamble(1) | msg_type(1) | node_id(1) | msg_id(1) | ttl(1) | topic(1) | pld_len(1) | payload(≤3) | crc8(1)]
최대 11바이트 — 64바이트 FIFO 제약을 크게 여유 있게 만족
```

- **version 제거**: 단일 버전 운용, 바이트 절약
- **topic 고정 1바이트**: 토픽 공간이 256개로 충분하며 topic_len 필드 불필요
- **node_id + msg_id 추가**: 브로커 없는 환경에서 중복 억제와 ACK 매칭에 필수
- **preamble 추가 (0xAB)**: LoRa 수신 시 노이즈 패킷 1차 필터링

## 12. 성공 기준 (Success Criteria)

- 수조 환경에서 익수자 교란을 발생시켰을 때, 파도와 구분하여 경보가 발생한다.
- 부표 B의 경보가 멀티홉 릴레이를 거쳐 수신국 대시보드에 표시된다.
- 평상시(파도만 있는 상태)에는 오탐 경보가 발생하지 않는다.
- LoRa 패킷이 64바이트 제약 안에서 토픽과 페이로드를 전달한다.
